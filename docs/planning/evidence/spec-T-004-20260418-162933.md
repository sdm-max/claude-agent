# SPEC T-004: package.json lint 스크립트 eslint 직접 호출로 변경

> 상태: NEW
> 우선순위: P0 (정식 게이트 1 lint 복구)
> 작성자 (원문): Reviewer (채팅 전달)
> 파일화 (transcription): Implementer (사용자 명시 지시, 2026-04-19)
> 유형: 빌드 스크립트 수정

---

## 1줄 요약
`package.json` 의 `"lint": "next lint"` 를 ESLint 직접 호출 (예: `"lint": "eslint src"`) 로 변경. Next.js 16 에서 `next lint` 가 deprecated 되어 `/lint` 경로 오인식 에러 발생하는 문제 해소.

## 완료 조건 (기계 검증 가능)
- [ ] `npm run lint` 실행 → "Invalid project directory provided" 에러 없음
- [ ] ESLint 가 `src/` 디렉토리를 대상으로 정상 실행
- [ ] 기존 `.eslintrc` / `eslint.config.*` 설정 그대로 적용 (프로젝트 규칙 유지)

## Scope
- **포함**: `package.json` 의 `scripts.lint` 값 한 줄 변경
- **제외**:
  - ESLint 설정 파일 수정
  - 기존 린트 에러 수정 (발견 시 별도 task)
  - 다른 스크립트(`build`, `dev`, `start`) 수정

## 가정
- 프로젝트에 ESLint 가 이미 설치되어 있음 (next.js 기본 의존성으로 포함)
- `eslint.config.*` 또는 `.eslintrc*` 파일 존재
- `src/` 디렉토리가 lint 대상 (프로젝트 구조상 `src-dir` 설정)
- **가장 위험한 가정**: 기존 `next lint` 가 주던 Next.js 특화 규칙이 일부 적용 안 될 수 있음 → 그 경우 별도 task 로 `eslint-config-next` 명시 설정

## Edge Case
1. ESLint 최상단 config 에 Next.js 규칙 없으면 기존 경고가 사라질 수 있음 (사용자가 원하면 `eslint-config-next` 를 config 에 추가하는 후속 task)
2. `src/` 외부 파일(예: `next.config.ts`) 은 lint 범위 밖 — 허용
3. ESLint cache 파일 생성되면 .gitignore 에 추가 필요 (이 SPEC 범위는 아님)

## 성공 판정 명령
```bash
# 에러 없이 실행되거나 lint 규칙 위반만 보여야 (not directory error)
npm run lint 2>&1 | grep -q "Invalid project directory" && echo "FAIL" || echo "PASS"
# 기대: PASS
```
