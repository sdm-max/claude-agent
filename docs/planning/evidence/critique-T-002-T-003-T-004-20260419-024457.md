## 1. SPEC 별 비판

### T-002 (hook line 36 `|| true`)

**강점**: 스코프 최소 (한 줄 변경). 회귀 위험 낮음.

**약점·우려**:
1. `|| true` 는 해당 pipe 전체의 **모든 실패**를 숨김. 예: `.claude/pipeline/` 디렉토리 자체가 사라진 경우도 silent. SPEC §"가장 위험한 가정" 에 적혀 있지만 **대안 검토 부족**.
   - 대안 1: `shopt -s nullglob; files=(.claude/pipeline/alerts/ALERT-*.md); ALERT_FILE="${files[-1]:-}"` — glob 자체를 null 로 처리해서 pipefail 우회 없음.
   - 대안 2: `ALERT_FILE=$(find .claude/pipeline/alerts -maxdepth 1 -name 'ALERT-*.md' -print 2>/dev/null | sort -r | head -1)` — find 는 no-match 도 exit 0.
   - 대안 중 bash-ism 자제 하려면 find 유리. 제안: SPEC 확장하여 alternatives 비교 후 최종 방법 명시.

2. 성공 판정 명령이 **불충분**:
   ```bash
   OUT=$(echo '{"tool_name":"Bash",...}' | bash .claude/hooks/e2e-before-commit.sh 2>&1)
   test -n "$OUT" || exit 1
   ```
   이건 "빈 output 이 아니면 PASS" 만 확인. Guard 0d 가 실제로 올바른 path 탐지하는지 검증 안 함 (ALERT 있을 때 block / 없을 때 pass).
   - 보강: ALERT 파일 있을 때/없을 때 두 케이스 모두 테스트 → ALERT 존재 시 block decision, 없을 때 통과 확인.

3. 이 hook 은 `set -e` / `pipefail` 외에도 다른 곳에 **같은 패턴** 쓰일 수 있음 (line 49 `ls -t ... diff-review-*.md | head -1`). T-002 범위가 line 36 만 → **line 49 도 동일 버그 가능성**. 별도 SPEC 로 확장 필요.

### T-003 (page.tsx Suspense 래핑)

**강점**: Next.js 16 표준 솔루션. 단일 파일 수정.

**약점·우려**:
1. **T-001 과 같은 파일 (src/app/templates/page.tsx)** 수정 — SPEC §"의존성" 에 애매하게 언급만 됨:
   > "T-001 먼저 merge or discard 결정 후 T-003 진행 권장"

   실제로 T-001 은 `blocked-spec-missing` + `deny` 로 commit 불가. **discard / merge 결정 없이 T-003 착수하면 T-001 diff 가 같은 수정분에 포함되어 의미가 섞임**. SPEC 이 명확히 한 가지 선택 강제 필요.

2. Suspense fallback 디자인 언급 없음:
   > "\<Suspense fallback={\<div>Loading...\</div>}\>"

   프로젝트 기존 로딩 패턴 (shadcn Spinner, Skeleton 등) 과 불일치. 눈에 띄는 UX 회귀 가능.

3. 성공 판정 명령이 **build exit 0 만 확인**. /templates 런타임 동작 (카드 렌더링, Apply Bar, 카테고리 필터) 이 여전히 동작하는지 검증 안 됨. e2e-scenarios.sh S1~S4 가 이를 커버할지 불확실.

4. **파일 구조 변경** 언급 모호:
   > "기존 export default function TemplatesPage() 를 내부 컴포넌트(예: TemplatesPageInner)로 이름 변경"

   기존 컴포넌트 이름 변경 시 혹시 storybook / 테스트 / 외부 참조 있으면 깨짐. 현재 확인 안 됨 (Implementer 자체 smoke 금지로 정식 grep 도 필요).

### T-004 (package.json lint 스크립트)

**강점**: 최소 변경 (1줄).

**약점·우려**:
1. **`eslint src` vs `eslint .`** 결정 미완:
   > "Reviewer 결정"

   Reviewer 가 결정 안 한 채 SPEC 전달 → Implementer 가 결정해야 하는 상황. 추측 금지 원칙 위반 위험.
   
   비교:
   - `eslint src`: `src/` 만 lint. `next.config.ts`, `drizzle/**` 는 제외. 빠름.
   - `eslint .`: 전체. `.gitignore` 기반 제외 필요. `next.config.ts` 도 커버.
   - → SPEC 명시 필요.

2. **ESLint 설정 파일 존재 확인** SPEC 에 없음. `eslint.config.*` / `.eslintrc*` 있어야 실행 가능. 없으면 T-004 끝나도 "ESLint not configured" 에러. **사전 조건 검증 미흡**.

3. **Next.js 특화 규칙 소실** 우려를 "별도 task 로 분리" 로 미룸 → 당장 `eslint-config-next` 확인 없으면 lint 후 기존 프로젝트의 Next.js 관련 린트 경고 (예: `<img>` → `<Image>`) 가 사라짐. 이걸 "후속 task" 로 미루는 건 위험.

4. `package.json` 수정은 `package-lock.json` 영향 없지만, 만약 ESLint 미설치면 `npm install eslint` 도 필요 → SPEC 범위 밖으로 빠짐.

## 3. 권장 순서 (Implementer 자체 의견)

기존 사용자 지시 "T-002 → T-003 → T-004" 는 합리적이지만 **T-001 처리 결정 먼저 필요**:

```
0.  Reviewer 결정: T-001 discard? 그러면 git checkout HEAD -- src/app/templates/page.tsx
    (→ T-003 착수 시 clean baseline)
    
1.  T-002 (hook) — 가장 안전 + commit 게이트 우선 해소
2.  T-004 (lint) — 독립, 빠름
3.  T-003 (Suspense) — T-001 결정 후에만, page.tsx 에 충돌 없는 상태에서
4.  (재작업) T-001 — 새 SPEC + 재구현 (현재 diff 폐기 후 재작성)
```

또는 T-001 을 살리는 경우:
```
0.  Reviewer: spec-T-001 본문 재작성 or 복원
1.  T-002 → T-004 → T-003 순서
2.  T-003 이 page.tsx 의 T-001 변경 위에 Suspense 추가 (1 파일 2 변경)
3.  T-001 + T-003 를 묶은 commit 승인
```

**후자가 코드 경제적**. 하지만 task 경계 명확성은 전자.
